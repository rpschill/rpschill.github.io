Side project · 2016–2017

projectr

A task management app built for one person — me. What happens when the developer and the user are the same person with ADHD and strong opinions about outlines.

Role
Designer, developer
Stack
AngularJS, JavaScript, Firebase
Hosted
GitHub Pages
Status
Legacy — live demo
The problem

I needed a task manager that thought the way I do.

I have ADHD, which means task management isn't optional — it's how I stay functional. I'd tried every app I could find, but none of them worked the way my brain works. They were either too simple, too complex, or organized around teams and collaboration when I just needed to manage my own work as a freelancer.

I was also a big fan of Workflowy — a minimalist, keyboard-navigable outline tool — and I wanted that same interaction model applied to actual project management. So I built it myself.

I also had a practical need: my job at the time required tracking how much time I spent on individual tasks. I wanted that built into the same tool I was already using to manage my work, not as a separate app.

What it does

Client → project → task → subtask.

projectr organizes work in a strict four-level hierarchy: clients contain projects, projects contain tasks, tasks contain subtasks. This maps directly to how freelance work is actually structured and keeps everything navigable without getting overwhelming.

The task list itself is keyboard-navigable in the style of an outline editor — you can create, nest, reorder, and complete tasks without touching the mouse. Each task can have a due date, notes, and attached file references.

The time tracking feature is built directly into individual tasks. Start a timer when you begin work, stop it when you're done. The elapsed time accumulates across sessions and rolls up to the project level so you can see where your hours are actually going.

Technical decisions

Keeping it as simple as possible for one developer.

Every technical choice I made was optimized for the same thing: let me focus on building features, not infrastructure. I was a solo developer building something for myself, and I wanted to spend my time on the parts that mattered.

AngularJS was my primary framework at the time and I knew it well. It gave me routing, two-way data binding, and a service-based architecture for free. All of the business logic lives in AngularJS services, which kept the components lean and the data layer testable.

Firebase Realtime Database meant I didn't have to write a backend. The data syncs in real time across sessions, user authentication is handled out of the box, and I didn't have to think about servers, deployments, or APIs. For a solo front-end developer, this was the right call.

GitHub Pages for hosting. Because Firebase handles the database, the entire app is static — just HTML, CSS, and JavaScript. Deploying meant committing code and pushing. That's it.

The result was a stack where I could go from idea to shipped feature in an afternoon without touching anything outside of front-end code.

Reflection

What I'd do differently today.

projectr was the first non-trivial application I'd ever built entirely on my own. That context matters: the choices I made were the right ones for where I was then. But I've spent nearly a decade building large-scale Angular applications since, and I'd approach it very differently now.

I'd use the current version of Angular rather than AngularJS. Modern Angular includes robust routing and reactivity out of the box, along with RxJS integration for handling the kind of complex, real-time data streams that a Firebase-backed app naturally produces. The reactive model would make the timer logic significantly cleaner.

I'd also reach for TypeScript from the start. The service layer in projectr would benefit enormously from typed interfaces — a Task, a Project, a Client as first-class types rather than plain JavaScript objects.

What I wouldn't change is the fundamental architecture: a front-end-only stack backed by a real-time cloud database, deployed as a static site. For this kind of single-developer personal tool, that's still the right call.
